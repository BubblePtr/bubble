# 基础 REPL 循环

第一章先介绍 Agent 的基础循环。这个循环是整个 Agent 系统的心脏——无论后续加上多复杂的记忆、上下文管理或自我进化能力，底层始终是这一层循环在驱动。理解并掌握它，就等于拿到了构建一切 Agent 能力的起点。

## 什么是 REPL ？

REPL = Read → Eval → Print → Loop，它是"读入 → 执行 → 输出 → 再等待输入"的交互循环。这个循环完美契合了我们与大模型交互的模式。

<ReplCycleAnimation />

### 这个循环，需要哪些对象？

1. **User**：用户，发起请求的人，也是最终接收回答的对象。
2. **Assistant**：大模型，负责理解意图、生成回答，并决定是否需要调用工具。
3. **Tool**：工具，Agent 可以调用的外部能力，比如读文件、执行命令等。
4. **Message**：聊天记录，承载整个对话过程，每一轮交互都会追加到其中。

现在我们把这些对象和这个循环连接上

## 认识 Tool Call

什么是 Tool Call（工具调用）？

大语言模型只能生成文本，无法真正读文件、跑命令、访问网络。但我们可以给它一份“工具菜单”，它就能在回复中表达“我想调用某个工具”的意图。有了 Tool Call 大模型就有了接触计算机系统的能力。

### Tool Call 的格式

模型回复中可能包含 tool call，以 OpenAI 格式为例：

```json
{
  "id": "fc_12345xyz",
  "call_id": "call_12345xyz",
  "type": "function_call",
  "name": "get_weather",
  "arguments": "{\"location\":\"Paris, France\"}"
}
```

三个关键字段：

- **id**：调用的唯一标识，返回结果时需要对应上。
- **name**：要调用哪个工具。
- **arguments**：传给工具的参数，JSON 字符串。

我们的程序需要做四件事：

1. 从模型回复中解析出 tool call。
2. 根据 `name` 找到对应工具，传入 `arguments` 执行。
3. 把结果包装成 `tool` 角色的消息，追加到聊天记录。
4. 带上更新后的聊天记录再次请求模型。

这就是 Agent Loop 中"Tool 分支"的逻辑。模型可能连续调用多个工具，直到收集够信息再给出最终回答。

### 如何定义一个 Tool？

模型需要知道有哪些工具可用，我们在请求中传入工具定义：

```json
{
  "type": "function",
  "name": "get_weather",
  "description": "Retrieves current weather for the given location.",
  "parameters": {
    "type": "object",
    "properties": {
      "location": {
        "type": "string",
        "description": "City and country e.g. Bogotá, Colombia"
      },
      "units": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "Units the temperature will be returned in."
      }
    },
    "required": ["location", "units"],
    "additionalProperties": false
  },
  "strict": true
}
```

三个关键的字段：

- **name**：工具名称，模型回复时引用。
- **description**：描述工具的能力和用途。
- **parameters**：参数的 JSON Schema，定义参数名、类型和是否必填。

description 写得越清晰，模型越能准确判断何时、如何使用工具——这是 Agent 调优的重要手段。

## 多轮调用与退出条件

### 多轮调用时消息历史的变化

每次工具调用都会往历史里追加两条消息：一条是模型的 assistant 消息（带 tool_calls），一条是执行结果的 tool 消息。下面是一次"调用两个工具、最终给出回答"的完整历史：

```
[system]    "You are a minimal coding agent..."
[user]      "读一下 Cargo.toml，然后把版本号写到 version.txt"
[assistant] tool_calls: [{ name: "read", args: { path: "Cargo.toml" } }]
[tool]      'name = "bubble"\nversion = "0.1.0"'
[assistant] tool_calls: [{ name: "write", args: { path: "version.txt", content: "0.1.0" } }]
[tool]      "ok"
[assistant] "已将版本号 0.1.0 写入 version.txt。"   ← 无 tool_calls，循环退出
```

每一轮都带着完整历史重新请求模型，模型才能知道前几步做了什么。

### 什么时候退出？

判断退出有两个信号，缺一不可。

**信号一：tool_calls 为空**

模型这轮没有发起任何工具调用，说明它已经准备好直接回答了。这是最直接的退出条件。

**信号二：API 返回的停止原因**

每次调用模型，响应里都会附带一个字段，说明这次是为什么停下来的：

| API | 字段名 | 继续循环 | 退出循环 |
|-----|--------|---------|---------|
| OpenAI | `finish_reason` | `"tool_calls"` | `"stop"` / `"length"` / `"content_filter"` |
| Anthropic | `stop_reason` | `"tool_use"` | `"end_turn"` / `"max_tokens"` / `"stop_sequence"` |

其中 `"length"` / `"max_tokens"` 是需要特别处理的情况——模型因为超出 token 限制而被截断，这轮回复是不完整的，既没有 tool call，也没有完整答案，必须作为错误退出，不能把截断内容当作正常回答返回给用户。

**最小实现 vs 健壮实现**

只检查 `tool_calls.is_empty()` 能覆盖绝大多数场景，这也是本章实现采用的方式。生产级实现还需要额外处理：

- `finish_reason == "length"` → 返回错误或触发上下文压缩
- 最大迭代次数限制 → 防止模型陷入无限工具调用循环
- Token 预算检查 → 主动在超限前截断

这些话题在后续章节（Context Engineering、Evolution）里会展开。

## 把循环写出来

理论讲完，看看代码长什么样。整个 agent 的骨架只有两层循环。

**外层：REPL 循环**

```rust
loop {
    // Read：读取用户输入
    let line = read_stdin()?;

    // Eval：跑一轮 agent
    let answer = run_agent_turn(&mut messages, &mut chat, &line).await?;

    // Print：输出结果
    println!("{answer}");

    // Loop：回到顶部等待下一条输入
}
```

这就是完整的外层循环。`messages` 在每轮之间保留，所以模型始终能看到完整的对话历史。

**内层：Tool Call 循环**

用户每发一条消息，agent 内部可能要跑好几轮才能给出最终回答：

```rust
loop {
    let assistant = chat(messages.clone()).await?;
    messages.push(assistant_message);

    if tool_calls.is_empty() {
        return Ok(final_answer); // 没有工具调用，直接返回
    }

    // 执行所有工具，把结果追加到历史
    for call in tool_calls {
        let result = execute(&call)?;
        messages.push(tool_message(call.id, result));
    }
    // 带着更新后的历史再次请求模型
}
```

两层循环叠在一起，就是一个完整的 coding agent。外层负责与用户交互，内层负责驱动模型完成单次任务——可能中途调用若干次工具，最终返回一个文本答案。

## Prompt

用下面这份 prompt 配合任意 coding agent，可以用你熟悉的语言实现本章的 REPL agent：

```
实现一个最小可运行的 REPL coding agent，要求：

1. 维护一个 messages 数组，包含 system / user / assistant / tool 四种角色
2. 外层是标准 REPL 循环：读取用户输入 → 调用 agent → 打印结果 → 等待下一条
3. 内层是 tool call 循环：请求模型 → 如有 tool call 则执行并追加结果 → 重复，
   直到模型返回纯文本为止
4. 工具集：bash（执行 shell 命令）、read（读文件）、write（写文件）
5. 系统提示词固定为："You are a minimal coding agent. Use tools when needed."
6. 使用 OpenAI 兼容接口，模型可配置

不需要配置文件、不需要日志、不需要错误重试，只要跑通核心循环即可。
```
